# Macro Contract Schema — Orchestrated Invariant Engine

A **contract** is a small, declarative description of how one macro
(`domain.name`) is allowed to behave. The Orchestrated Invariant Engine reads
contracts and drives every headless-safe macro through three adversarial
vectors (seed / fuzz / invariant). Contracts are how a human pins *intent*; the
engine turns that intent into pass/fail signal against the **real** `runMacro`.

There are two tiers of contract, and you almost never write the first one by
hand:

1. **Derived baselines** (`content/contracts/derived/<domain>.json`) — auto-
   generated by `scripts/contracts/derive-contracts.mjs`. One JSON file per
   domain, holding one entry per `(domain, macro)` pair. Inputs come from the
   macro's `spec.paramSchema` when present (else empty). Invariants are the
   universal floor every macro must clear. **Do not hand-edit these** — they are
   regenerated and overwritten.

2. **Overrides** (`content/contracts/overrides/<domain>.<macro>.json`) — the
   only files you author by hand. An override is *additive/merge-on-top* of the
   derived baseline: it tightens inputs, adds domain-specific invariants, and
   supplies real seed `fuzz_cases`. You only write overrides for macros whose
   correctness you actually care to pin beyond the universal floor.

## Contract shape

```jsonc
{
  "macro_id": "domain.name",          // required — the canonical (domain.macro) id
  "domain":   "domain",               // required — the macro's domain
  "name":     "name",                 // optional — derived from macro_id if omitted

  // INPUTS — a declarative param schema, same shape as
  // server/lib/macro-param-schema.js. Used to (a) synthesize a minimal-valid
  // input for the V3 invariant vector and (b) build the malicious V2 payload
  // by poisoning numeric / string params. Overrides MERGE field-by-field onto
  // the derived baseline.
  "inputs": {
    "count":  { "type": "number", "min": 1, "max": 100 },
    "title":  { "type": "string", "required": true, "maxLength": 200 },
    "mode":   { "type": "string", "enum": ["a", "b"] }
  },

  // INVARIANTS — JS boolean expressions over two free variables: `input` and
  // `output`. Evaluated by server/lib/invariant-eval.js (compile-once, never
  // throws). A `false` result is a V3 violation. These MERGE (append + dedupe)
  // onto the derived universal invariants.
  "invariants": [
    "typeof output === 'object' && output !== null",
    "output.count >= 0",
    "output.count <= input.max"
  ],

  // FUZZ_CASES — concrete seed vectors (V1). Each runs its `input` through the
  // real runMacro and asserts every key in `expect` deep-equals the macro's
  // returned value at that key (shallow key match, deep value compare).
  "fuzz_cases": [
    { "input": { "count": 3 }, "expect": { "ok": true } },
    { "input": { "count": -1 }, "expect": { "ok": false } }
  ]
}
```

### Field reference

| Field         | Required | Meaning |
|---------------|----------|---------|
| `macro_id`    | yes      | `"domain.name"` — the registry key. |
| `domain`      | yes      | Macro domain (the `domain` half of `macro_id`). |
| `name`        | no       | Macro name; inferred from `macro_id` when absent. |
| `inputs`      | no       | Param schema (`{type,min?,max?,required?,enum?,maxLength?}` per field). |
| `invariants`  | no       | Array of JS boolean expression strings over `(input, output)`. |
| `fuzz_cases`  | no       | Array of `{ input, expect }` seed vectors for V1. |
| `_derived`    | (auto)   | Present in derived files; marks the baseline so overrides can tell. |

### Input param rule shape

Mirrors `server/lib/macro-param-schema.js`:

- `type`: `"string" | "number" | "boolean" | "array" | "object"`
- `required`: boolean
- `min` / `max`: numeric bounds (numbers only)
- `enum`: array of allowed values
- `maxLength`: string length cap

### The three vectors (what the engine does with a contract)

- **V1 SEED** — runs each `fuzz_cases[].input` through `runMacro` and checks the
  returned object matches every key in `expect`. Seeds are the human-pinned
  "this exact input must yield this exact shape" assertions.
- **V2 FUZZ** — synthesizes a *malicious* payload from `inputs` (numeric params
  poisoned with `NaN`/`Infinity`/`1e308`/`-1`, strings with an XSS probe, plus a
  deeply-nested object) and calls `runMacro`. **Pass** if the macro returns
  `{ok:false}` or throws a *caught* guard error. **Fail** only on an uncaught
  crash that aborts the runner, or `{ok:true}` returned over poisoned numeric
  input. (Extra/ignored fields are tolerated — most macros legitimately ignore
  them.)
- **V3 INVARIANT** — runs a minimal-valid input (param `min`s, or `{}`) and
  evaluates every `invariants[]` expression. A `false` is a violation.

### Universal (auto-derived) invariants

Every derived baseline carries this floor, so even a macro with no override is
checked:

1. `output !== null && output !== undefined` — must return *something*.
2. `typeof output === 'object'` — the envelope is always an object.
3. `(typeof output.ok === 'boolean') || (output !== null && typeof output === 'object')`
   — either a `{ok:boolean}` envelope OR a plain data object (both are accepted
   macro return shapes in this codebase).

### Authoring an override (the normal workflow)

1. Pick a macro you want to pin, e.g. `accounting.trialBalance`.
2. Create `content/contracts/overrides/accounting.trialBalance.json`.
3. Add real `inputs`, domain `invariants`, and at least one `fuzz_case`.
4. Re-run `node scripts/contracts/derive-contracts.mjs` (merges your override
   onto the baseline) then `node scripts/macro-assassin.mjs`.

See `content/contracts/_example.json` for a complete worked override.

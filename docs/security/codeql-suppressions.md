# CodeQL Suppression Ledger

Sprint 18.4 — audit ledger for every CodeQL query-ID exclusion in
`.github/workflows/codeql.yml`.

**Last updated:** 2026-05-11
**Reviewer:** Maintainer (solo)
**Re-review cadence:** every 90 days, via
`.github/workflows/codeql-exclusions-review.yml` cron

**The contract:** every entry in the workflow's `query-filters` block has a
row in this table. Any change to that block (add / remove / scope-change)
requires a matching ledger update in the same commit. The
`platinum-codeql-drift.test.js` test in `server/tests/` enforces the
"if you excluded X, no new X may appear outside the audit allowlist"
invariant at PR time — even though CodeQL is silent on the category.

## How to read this table

| Field | Meaning |
|---|---|
| **Query ID** | Exact CodeQL rule ID (`js/<rule-name>`) |
| **Scope** | `repo` = excluded codebase-wide; `path:<glob>` = excluded only for matching files |
| **Audit date** | When the manual audit verified every flagged site is FP |
| **Allowlist** | Files where the pattern legitimately appears (audit-verified) |
| **Drift gate** | The test in `platinum-codeql-drift.test.js` that catches new violations |
| **Re-audit trigger** | What event invalidates this suppression and forces a re-check |

## Active suppressions

### 1. `js/insufficient-password-hash`

| | |
|---|---|
| Scope | repo |
| Audit date | 2026-04-30 |
| Allowlist | `server.js#generateApiKey`, all `crypto.randomBytes(32)` token mints |
| Drift gate | N/A — bcrypt is positively asserted for `users.password_hash` writes |
| Re-audit trigger | Any new low-entropy secret storage (passwords, OTP codes, recovery codes) lands |

**Justification:** Concord's API keys are 256-bit random tokens generated by
`crypto.randomBytes(32)`. SHA-256 is the correct hash for high-entropy random
input. Slow KDFs (bcrypt / scrypt / argon2) only protect low-entropy
passwords against brute-force, which is irrelevant for 256-bit random input.

| | |
|---|---|
| Scope | repo |
| Audit date | 2026-05-11 (Explore-agent survey of 1,184 server files) |
| Allowlist | Every user-input regex site has a length cap (`slice(0, N)`, `MAX_*_LEN`, `clip(s)`) |
| Drift gate | `platinum-codeql-drift.test.js#regex-on-user-input sites have a length cap nearby` |

**Justification:** CodeQL's taint tracker can't follow `.slice()` /

### 7. `js/sql-injection`

| | |
|---|---|
| Scope | repo |
| Audit date | 2026-05-11 |
| Allowlist | All `db.prepare()` sites — better-sqlite3 parameterized prepares |
| Drift gate | `platinum-codeql-drift.test.js#db.prepare() interpolations are server-constructed clauses, not user input` |
| Re-audit trigger | `req.body.*` / `req.params.*` / `req.query.*` interpolated directly into a `db.prepare()` template string |

catches the actually-dangerous pattern of user-input interpolation.

## Kept-enabled queries (high blast radius, FP noise tolerated)

These queries are NOT suppressed. Any new flag goes through manual triage.

- `js/path-injection` (filesystem traversal — even with `containedPath()` helper, every new fs op is reviewed)
- `js/server-side-request-forgery` (internal-network exfil)
- `js/xss` family (output-context tampering)
- `js/sensitive-data-exposure`
- `js/incomplete-url-substring-sanitization`

If you find yourself wanting to suppress one of these, **stop and re-audit**.
The blast radius doesn't justify suppression — fix the source instead.

## Re-review protocol

When the 90-day cron opens an issue (or any new code lands that matches
a re-audit trigger):

1. Re-run the Explore-agent survey for each suppressed category. Confirm
   the allowlist file set hasn't grown silently.
2. For each new file matching a forbidden-pattern grep, classify:
   real bug → fix at source; verified FP → add to allowlist with comment.
3. Update this ledger's "Audit date" + any allowlist changes.
4. Re-run `tests/platinum-codeql-drift.test.js` — must pass.
5. Close the issue with a 1-paragraph audit note linking to the commit.

If a suppression can't be re-verified (audit team changed, allowlist
files grew suspiciously, etc.), the suppression must be REMOVED until
the team can re-audit.

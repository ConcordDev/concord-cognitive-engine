# Lens Coverage Audit

Generated by cross-referencing 254 backend domain files (1,106 macros)
against 231 lens frontends. Counts how many backend macros each lens
actually wires to its UI vs. how many sit unused.

## Headline numbers

- **254** backend domain files at `server/domains/*.js`
- **1,106** macros registered (~6 per domain average)
- **231** lens dirs at `concord-frontend/app/lenses/*/`
- **115** lenses (~50%) call at least one backend macro
- **93** lenses (40%) have a backend domain but call **zero** macros — biggest opportunity bucket
- **46** lenses (20%) have no backend domain at all — these need backend before UI
- **48** lenses (~20%) at full coverage — most macros wired
- **44** lenses (~19%) at partial coverage — backend exists, some macros wired but more available

## How this was measured

- **Backend macros**: grep `server/domains/*.js` for `registerLensAction("<domain>", "<name>", …)` and count distinct `(domain, name)` pairs.
- **UI wiring**: for each lens, scan `app/lenses/<lens>/page.tsx` + `components/<lens>/**/*.tsx` for `useRunArtifact("<domain>")` (which backend domain does the lens target) and `action: "<name>"` (which macros it actually calls).
- **Coverage**: `macros_wired / macros_available`.

See `/tmp/lens-coverage.json` for the full machine-readable dump (or regenerate via `python3 docs/scripts/lens-coverage.py`).

## What this audit does NOT measure

- **Wiring depth.** A lens that calls 1 macro 1 time vs. 1 macro 100 times scores the same.
- **UI polish.** "Has a button that calls the macro" is the bar, not "feels like the paid app."
- **Backend correctness.** Macros are counted as available regardless of whether they actually work.
- **Cross-domain calls.** `crisis-ops` calling `crisis.declare` looks like "no-backend" to this script because the domain isn't named after the lens. Spot-check the no-backend bucket before assuming work is needed.
- **Generics.** `useArtifacts` / `useLensData` calls bypass action-routing entirely — those lenses look "unwired" here but may be fully functional via direct REST.
